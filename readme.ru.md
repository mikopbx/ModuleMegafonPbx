## ModuleMegafonPbx — загрузка истории звонков из MegaPBX в MikoPBX

Коротко:

- **Назначение**: импорт истории звонков из MegaPBX в CDR MikoPBX.
- **Дальше**: из MikoPBX историю можно выгружать в 1С:Предприятие (см. `getHistory.epf`).
- **Ограничение**: интеграция работает только с MegaPBX; модуль читает историю и не управляет звонками.

### Требования
- Доступ к API MegaPBX (CRM API) и актуальный токен/учетные данные.
- Установленный MikoPBX с правами установки и запуска модулей.

### Установка
1. В веб‑интерфейсе MikoPBX перейдите: Модули → Маркетплейс модулей → Загрузить новый модуль.
2. Выберите архив модуля `ModuleMegafonPbx` и дождитесь установки.
3. Активируйте модуль.

### Настройка
В настройках модуля укажите:
- **Адрес АТС Мегафон**: например `vats.megafon.ru`.
- **Ключ для авторизации API** MegaPBX.
- **Какой номер отображать в истории** - внутренний или мобильный номер сотрудника.
- **Сдвиг времени** - позволяет скорректировать время на указанное количество часов.
- **CRM-токен** — секрет, которым ВАТС МегаФон подписывает входящие webhook-события
  (должен совпадать с токеном, заданным в личном кабинете ВАТС, см. ниже).

Сохраните изменения.

Скрипт использует сохраненные в модуле настройки подключения и период синхронизации.

### Как это работает
- Модуль обращается к API истории MegaPBX и получает события за период.
- Записи добавляются в CDR MikoPBX; предусмотрены механизмы для предотвращения дублей.
- Модуль работает только на чтение и не изменяет данные в MegaPBX.

### Входящий webhook от ВАТС МегаФон

Помимо периодического пулла, модуль публикует REST-эндпоинт, на который
ВАТС МегаФон может отправлять события (реалтайм-уведомления о звонках,
финальные записи с ссылкой на mp3, поиск контакта для попап-карточки):

```
POST https://<адрес-pbx>/pbxcore/mega-pbx/event
Content-Type: application/x-www-form-urlencoded   (или application/json)
```

Авторизация — через поле `crm_token` в теле запроса (должно совпадать
с **CRM-токеном** из настроек модуля).

Поддерживаемые значения `cmd`:
- `event`   — состояние звонка (INCOMING / ACCEPTED / COMPLETED / …);
              если установлен и настроен `ModuleCTIClient`, событие
              отправляется в 1С через SOAP, иначе — пишется в файловый лог.
- `history` — финальная запись о звонке со ссылкой на mp3; принимается тихо
              (`200 ok`), так как историю и записи уже подгружает cron-воркер
              `bin/synchCdr.php` через `/crmapi/v1/history/json`.
- `contact` — поиск клиента по номеру для всплывающей карточки на IP-телефоне.
              Возвращает `{"contact_name": "..."}`, если установлен
              `ModuleCTIClient` и его CRM-демон отвечает; иначе `{}`.

Настройка webhook на стороне МегаФон:
1. Личный кабинет ВАТС → Интеграции → CRM API.
2. **Адрес CRM**: `https://<адрес-pbx>/pbxcore/mega-pbx/event`.
3. **Токен**: тот же, что в настройках модуля.

Лог пишется с ротацией (40 МБ × 9 файлов) в:
```
/storage/usbdisk1/mikopbx/log/ModuleMegafonPbx/EventController.log
```

### Безопасность и firewall

Webhook-эндпоинт — **публичный** (нет Bearer/cookie авторизации, только
`crm_token` в теле). Как минимум:

- **Только HTTPS.** По HTTP `crm_token` уйдёт открытым текстом и его
  перехватят на любом промежуточном узле.
- **Ограничьте источник на firewall.** ВАТС МегаФон отправляет события
  с известного адреса — например, с боевой ВАТС трафик идёт с
  `193.201.230.155`. Настройте файрвол или nginx-allowlist так, чтобы
  путь `/pbxcore/mega-pbx/*` принимал запросы только с этого адреса
  (и других IP, документированных вашим оператором МегаФон —
  актуальный список лучше уточнить в техподдержке).
- **Используйте длинный случайный `crm_token`** (например, UUIDv4)
  и периодически его меняйте.
- **Следите за записями `auth_fail`** в `EventController.log` —
  частые попытки с неверным токеном указывают на брутфорс.

### Экспорт в 1С:Предприятие
- Историю звонков, импортированную в MikoPBX, можно выгружать в 1С стандартными средствами.
- В репозитории присутствует обработка `getHistory.epf` для загрузки истории из MikoPBX в 1С.
